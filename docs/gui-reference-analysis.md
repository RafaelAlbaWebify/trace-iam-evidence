# TRACE GUI visual reference analysis

## Reference set received

The user supplied the complete image set after the initial package audit. The images cover two visual families:

1. **Production Suite** — persistent light sidebar, compact page header, restrained metric cards, table-first operational pages, subtle borders, sparse semantic colour, compact actions, modal and component-system examples.
2. **Dashforge** — dense multi-panel dashboards, richer KPI composition, activity panels and stronger analytical overview rhythm.

The approved TRACE concept mock is also retained as a directional reference, but unsupported content in that mock is explicitly excluded.

## Controlling visual decision

TRACE will use:

- **Production Suite as the primary structural and component reference**;
- **Dashforge only as a secondary reference for dashboard rhythm and information density**;
- **TRACE's verified domain model as the sole source of terminology, fields, metrics, actions and workflow composition**.

## Patterns adopted

- persistent operational navigation;
- calm light content canvas;
- compact page titles and page-level context;
- restrained bordered surfaces with limited shadows;
- metric strips using real workload counts;
- table-oriented case, evidence and run records;
- compact filters and actions;
- restrained semantic status pills;
- consistent dialogs, fields, buttons, empty states and errors;
- responsive navigation that remains keyboard accessible.

## Patterns explicitly rejected

TRACE will not import:

- revenue, shipment, production, sales or customer metrics;
- user avatars, investigators, assignees or performance rankings;
- tasks, approvals or collaboration workflows;
- artificial progress percentages;
- decorative charts without supported TRACE data;
- live monitoring or remediation affordances;
- generic admin-template terminology.

## TRACE-specific adaptation

The design must foreground:

- persisted investigation identity;
- scenario and lifecycle status;
- case priority and evidence readiness;
- provenance, reliability and validation;
- supporting, contradicting and missing evidence;
- deterministic rule identity and uncertainty;
- immutable runs, comparison and append-only timeline;
- local-first, redacted and read-only safety boundaries.

## Pilot implication

The operational dashboard remains the pilot. It should combine the Production Suite shell and table discipline with the more deliberate summary rhythm visible in Dashforge, while using only existing TRACE dashboard counts and case fields.
