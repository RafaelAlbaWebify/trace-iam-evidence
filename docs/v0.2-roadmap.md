# TRACE v0.2 — Investigation Workbench roadmap

## Product standard

TRACE v0.2 is not a demonstration layer over isolated analysis forms. It is a real, local-first support-engineering investigation workbench. Every capability must operate on persisted cases, preserve evidence provenance and immutable history, enforce safety boundaries in the backend, and pass automatic browser and cross-platform proof before merge.

## Milestone 8 — Real case management

### 8.1 Persisted investigation workspace — complete

Deliverables:

- operator-created investigations with server-generated IDs;
- persisted draft state before the first analysis;
- case title and scenario selection;
- active-case workspace and visible case identity;
- existing cases reopened from the case register;
- scenario-specific analysis locked to the active case;
- archive/reopen protection;
- immutable runs and exports attached to the generated case ID;
- server-side rejection of scenario, title, and archive-state mismatches.

Exit proof:

- all three browser workflows create real persisted cases;
- generated IDs match the documented `trace-<12 hex characters>` contract;
- analysis, history and exports use the generated ID rather than fixed demonstration IDs;
- Conditional Access browser proof archives and reopens the same case;
- backend and frontend checks pass on Ubuntu and Windows;
- release publisher and Ubuntu/Windows release packs remain green.

### 8.2 Investigation lifecycle and case metadata — complete

Deliverables:

- priority classification: low, normal, high and critical;
- public-safe external case reference;
- concise redacted case summary;
- operational metadata captured during case creation;
- active-case metadata editing without changing case identity or immutable runs;
- priority-aware active case and case register presentation;
- backward-compatible loading of older persisted snapshots;
- explicit lifecycle transition contract;
- backend rejection of invalid lifecycle jumps;
- metadata preservation through all three analysis routes;
- archive protection for edits and analysis;
- archive and reopen restore the exact prior lifecycle state rather than forcing a case backward.

Exit proof:

- metadata and lifecycle API tests pass on Ubuntu and Windows;
- frontend tests prove metadata intake and editing;
- Chromium proves create, triage, edit, analyze, review, archive, reopen and export;
- existing Conditional Access, resource-assignment and Guest/B2B workflows remain green;
- release publisher and Ubuntu/Windows release packs remain green;
- immutable lifecycle events remain correctly reserved for Milestone 11 rather than simulated prematurely.

## Milestone 9 — Evidence workspace

### 9.1 Persisted evidence model and API — complete

Deliverables:

- multiple evidence items per persisted case;
- evidence kind, source, collection time, subject, resource and redacted excerpt;
- reliability classification and redacted operator notes;
- explicit validation timestamps;
- duplicate evidence ID rejection within each case;
- archived-case protection;
- deletion blocked after immutable analysis runs exist;
- evidence edits invalidate the affected validation and return fully validated cases to draft;
- automatic transition to `evidence_validated` only after every evidence item is validated;
- backward-compatible loading of older evidence snapshots.

Exit proof:

- API tests prove add, list, update, validate and delete behavior;
- multiple items persist without loss;
- duplicate, archive and immutable-run protections are enforced by the backend;
- Ubuntu and Windows backend checks pass;
- existing frontend and Chromium workflows remain green;
- Ubuntu and Windows release packs remain green.

### 9.2 Operator evidence inventory — complete

Deliverables:

- modular active-case evidence workspace rather than additional monolithic application logic;
- case-bound evidence loading when investigations are created or reopened;
- evidence type, source, subject, resource, redacted excerpt, reliability and redacted notes intake;
- evidence inventory summary with item, validation and case-state counts;
- preview cards showing provenance, reliability and validation state;
- per-item validation controls;
- pre-analysis evidence deletion;
- archive and immutable-run protection reflected in disabled operator controls;
- responsive presentation and explicit empty states;
- active case and case register refresh after evidence changes.

Exit proof:

- frontend tests prove case creation, evidence addition and validation-driven lifecycle changes;
- Chromium creates a persisted case, adds provenance-rich evidence, validates it and observes `evidence_validated`;
- Chromium executes analysis and proves that evidence deletion is disabled after the immutable run exists;
- Conditional Access, resource-assignment and Guest/B2B browser workflows remain green;
- backend and frontend checks pass on Ubuntu and Windows;
- release publisher and Ubuntu/Windows release packs remain green.

### 9.3 Immutable evidence snapshots for analysis runs — complete

Deliverables:

- populated case inventories remain intact when scenario analysis executes;
- existing case evidence must be fully validated before supporting a run;
- backward-compatible first-run support for legacy scenario forms without a prebuilt inventory;
- every immutable JSON report embeds the exact redacted evidence set used by the run;
- every immutable Markdown report includes an evidence snapshot section;
- snapshots preserve evidence ID, type, source, reliability, capture time and validation time;
- retention mode is applied independently to the immutable snapshot;
- older stored reports without snapshots remain readable;
- subsequent live-evidence changes cannot alter an earlier run snapshot.

Exit proof:

- persistence tests prove append-only evidence snapshots and backward-compatible reloads;
- regression proof changes the live case inventory after Run 1 and confirms Run 1 remains unchanged;
- Chromium confirms exported JSON and Markdown identify the validated case evidence used by the run;
- Conditional Access, resource-assignment and Guest/B2B browser workflows remain green;
- backend and frontend checks pass on Ubuntu and Windows;
- release publisher and Ubuntu/Windows release packs remain green.

## Milestone 10 — Structured findings workspace

- finding cards with severity, confidence and rule version;
- supporting, contradicting and missing evidence views;
- separate safe checks, non-actions, limitations and unresolved questions;
- filtering and export while preserving raw JSON and Markdown.

## Milestone 11 — Investigation timeline

- system events for case creation, evidence changes, analyses, status changes and exports;
- redacted operator notes;
- immutable generated events and clearly attributed manual notes;
- chronological and newest-first views.

## Milestone 12 — Run comparison

- compare any two runs in a case;
- added, resolved and changed findings;
- confidence, evidence and ruleset differences;
- explicit explanation of whether changes came from evidence or rule versions;
- comparison report export.

## Milestone 13 — Search and operational dashboard

- search by title, redacted case reference and generated ID;
- filters for status, scenario, findings and date;
- operational workload cards without vanity metrics;
- recently updated and waiting investigations.

## Milestone 14 — Local product packaging

- one-command Windows start, stop and diagnostics workflow;
- prerequisite checks, migrations and service startup;
- configurable local data directory;
- SQLite backup and restore;
- portable review ZIP generated directly in Downloads without automatically opening the folder.

## Milestone 15 — Verified v0.2.0 release

- complete case-to-evidence-to-analysis-to-comparison browser workflow;
- multiple immutable runs and timeline events;
- search, reopen, export, backup and restore proof;
- Ubuntu and Windows CI and release packs;
- controlled Windows review;
- stable `v0.2.0` publication from the verified merge commit.

## Deferred beyond v0.2

Live Microsoft Graph connectivity, credential storage, automated remediation, direct access or policy changes, multi-user cloud hosting and AI-generated root-cause claims remain out of scope until the local investigation model and security design are mature and separately approved.
