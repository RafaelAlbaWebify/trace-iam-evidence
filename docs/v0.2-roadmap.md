# TRACE v0.2 — Investigation Workbench roadmap

## Product standard

TRACE v0.2 is not a demonstration layer over isolated analysis forms. It is a real, local-first support-engineering investigation workbench. Every capability must operate on persisted cases, preserve evidence provenance and immutable history, enforce safety boundaries in the backend, and pass automatic browser and cross-platform proof before merge.

## Milestone 8 — Real case management — complete

### 8.1 Persisted investigation workspace — complete

- operator-created investigations with server-generated IDs;
- persisted draft state before the first analysis;
- case title and scenario selection;
- active-case workspace and visible case identity;
- existing cases reopened from the case register;
- scenario-specific analysis locked to the active case;
- archive/reopen protection;
- immutable runs and exports attached to the generated case ID;
- server-side rejection of scenario, title, and archive-state mismatches.

### 8.2 Investigation lifecycle and case metadata — complete

- priority classification: low, normal, high and critical;
- public-safe external case reference and redacted case summary;
- active-case metadata editing without changing case identity or immutable runs;
- backward-compatible persisted snapshots;
- explicit backend-enforced lifecycle transitions;
- archive and reopen restore the exact prior lifecycle state.

## Milestone 9 — Evidence workspace — complete

### 9.1 Persisted evidence model and API — complete

- multiple evidence items per case;
- evidence type, provenance, collection time, reliability, notes and validation timestamps;
- duplicate, archived-case and immutable-run protection;
- validation-driven case readiness;
- backward-compatible loading.

### 9.2 Operator evidence inventory — complete

- modular active-case evidence workspace;
- case-bound loading, creation, preview, validation and pre-analysis deletion;
- provenance and reliability presentation;
- disabled controls where archive or immutable history prohibits changes;
- Chromium proof across all three scenarios.

### 9.3 Immutable evidence snapshots for analysis runs — complete

- validated case inventories remain intact during analysis;
- every immutable JSON and Markdown report captures the exact evidence set used;
- retention mode applies independently to run snapshots;
- old reports remain readable;
- later live evidence changes cannot alter earlier runs.

## Milestone 10 — Structured findings workspace — complete

- finding cards with severity, confidence, rule ID and version;
- supporting, contradicting and missing evidence views;
- separate safe checks, non-actions and limitations;
- severity and confidence filtering;
- raw JSON and Markdown inspection and immutable exports;
- defensive parsing and responsive presentation.

## Milestone 11 — Investigation timeline — complete

- append-only system events for case creation, evidence changes, analyses, lifecycle changes and exports;
- redacted operator notes with attribution;
- chronological and newest-first views;
- archived-case note protection;
- explicit report-export audit actions without mutating report GET requests.

## Milestone 12 — Run comparison — complete

- compare any two immutable runs in a case;
- added, resolved and changed findings;
- confidence, evidence and ruleset differences;
- deterministic finding identity and duplicate protection;
- comparison workspace and JSON export;
- two-run Chromium proof.

## Milestone 13 — Search and operational dashboard — complete

Deliverables:

- search by generated ID, title, redacted external reference and redacted summary;
- status, scenario, priority and archived-state filters;
- deterministic ordering by latest immutable timeline activity;
- workload counts for active, waiting-for-evidence, ready-for-analysis, under-review, critical-active and archived cases;
- case opening through generated investigation IDs;
- no inferred SLA, productivity, ownership or unsupported risk metrics;
- serialized shared database migration for safe concurrent first-launch requests.

Exit proof:

- backend service tests cover search, filters, workload counts and ordering;
- concurrency regression proves simultaneous repository initialization migrates once;
- frontend tests cover filtering and generated-ID case opening;
- Chromium searches by redacted case reference, opens the exact persisted case and retains dashboard evidence;
- Conditional Access, resource-assignment and Guest/B2B browser workflows remain green;
- backend and frontend checks pass on Ubuntu and Windows;
- release publisher and Ubuntu/Windows release packs remain green.

## Milestone 14 — Local product packaging

- one-command Windows start, stop and diagnostics workflow;
- prerequisite checks, migrations and service startup;
- configurable local data directory;
- SQLite backup and restore;
- portable review ZIP generated directly in Downloads without automatically opening the folder.

## Milestone 15 — Verified v0.2.0 release

- complete case-to-evidence-to-analysis-to-comparison browser workflow;
- multiple immutable runs and timeline events;
- search, reopen, export, backup and restore proof;
- Ubuntu and Windows CI and release packs;
- controlled Windows review;
- stable `v0.2.0` publication from the verified merge commit.

## Deferred beyond v0.2

Live Microsoft Graph connectivity, credential storage, automated remediation, direct access or policy changes, multi-user cloud hosting and AI-generated root-cause claims remain out of scope until the local investigation model and security design are mature and separately approved.
